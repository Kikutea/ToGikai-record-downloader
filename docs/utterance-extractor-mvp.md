# 発言抽出ツール MVP 仕様

## 1. 目的

東京都議会会議録サイト `https://www.record.gikai.metro.tokyo.lg.jp` の会議録ページから、
さんのへあや議員に関係する発言を抽出し、
公式サイト `https://sannohe-aya.com/archive/shitsumon/` に掲載しやすい形式へ整形する。

MVP の狙いは、現在の以下の手作業を最小化すること。

- 会議録ページを開く
- 該当会議を探す
- さんのへあや議員の発言箇所を探す
- 必要に応じて答弁も含めてコピペする
- WordPress に貼れる形へ整形する


## 2. MVP の範囲

MVP では以下のみ対応する。

- 会議録本文ページの URL を入力して処理する
- さんのへあや議員の発言を検出する
- 2 種類の抽出モードを提供する
  - `質疑全文`: 本人発言と、それに続く答弁をまとめて抽出
  - `本人発言のみ`: 本人の発言ブロックのみ抽出
- WordPress 貼り付け用 HTML を出力する
- あわせて Markdown / プレーンテキストも出力する
- タイトル、日付、会議名、カテゴリ候補を自動生成する

MVP では以下は対象外とする。

- 東京都議会サイト内の自動検索 UI
- 複数会議の一括処理
- WordPress への自動投稿
- 新着会議録の自動巡回
- AI による要約生成


## 3. 想定ユーザー

- さんのへあや議員本人
- 事務所スタッフ
- 公式サイト更新担当者


## 4. ユースケース

### 4.1 基本フロー

1. 担当者が東京都議会会議録の本文ページを開く
2. URL をツールに貼り付ける
3. 抽出モードを選ぶ
4. ツールが本文を解析し、プレビューを表示する
5. 担当者が内容を確認する
6. `WordPress 用 HTML をコピー` を押す
7. WordPress の本文欄に貼り付ける

### 4.2 対象 URL の例

- `https://www.record.gikai.metro.tokyo.lg.jp/100000?Template=document&Id=19676`
- `https://www.record.gikai.metro.tokyo.lg.jp/577167?Id=19676&Template=document`

`Id` を持つ本文ページを対象とする。


## 5. 画面仕様

### 5.1 MVP 画面

単一画面で十分。

- 入力欄
  - 会議録 URL
- オプション
  - 抽出モード
    - `質疑全文`
    - `本人発言のみ`
  - 除外オプション
    - `議長・委員長の進行発言を除外`
    - `拍手・起立などの地の文を除外`
- 出力
  - タイトル
  - 日付
  - 会議名
  - カテゴリ候補
  - 本文プレビュー
  - WordPress 用 HTML
  - Markdown
  - プレーンテキスト
- 操作
  - `抽出`
  - `HTML をコピー`
  - `Markdown をコピー`
  - `テキストをコピー`


## 6. 入力仕様

### 6.1 必須入力

- 東京都議会会議録本文ページの URL

### 6.2 任意入力

- 抽出モード
- ノイズ除外設定

### 6.3 バリデーション

以下の場合はエラーとする。

- URL が空
- ドメインが `record.gikai.metro.tokyo.lg.jp` ではない
- `Template=document` または `Id` が含まれない
- 本文ページの取得に失敗した
- さんのへあや議員の発言が 1 件も見つからない


## 7. データ取得仕様

### 7.1 取得元

- 東京都議会会議録検索サイトの本文ページ HTML

### 7.2 利用する要素

会議録本文ページには以下の情報が含まれる前提で処理する。

- 文書タイトル
  - 例: `令和７年第１回定例会(第７号) 本文`
- 日付
  - 例: `2025-03-28`
- 発言一覧
  - `VoiceID[]`
  - 発言者名
  - `data-voice_code`
- 本文ブロック
  - `li.voice-block`
  - `div.voice_text`
  - `p.voice__text`

MVP では API は使わず、HTML をパースする。


## 8. 発言検出ルール

### 8.1 本人判定

以下のいずれかに一致した発言者を「本人」と判定する。

- `さんのへあや`
- `さんのへ委員`
- `さんのへあや委員`
- `◯十一番（さんのへあや君）`
- `◯◯番（さんのへあや君）`

実装上は、以下のような緩い正規化を行う。

- 全角半角差を吸収
- 余分な空白を削除
- `◯`、議席番号、括弧書きの差異を許容
- `さんのへあや` をキーに部分一致できるようにする

### 8.2 ノイズ判定

以下は原則として除外候補とする。

- 議長、委員長、書記などの進行発言
- `〔賛成者起立〕` のような地の文
- `──────────` のような罫線
- 末尾の `（拍手）`

ただし、`質疑全文` モードでは、本人発言の前後関係を崩さない範囲で進行発言を残せるように内部フラグは持つ。
初期設定では除外する。


## 9. 抽出ロジック

### 9.1 本人発言のみ

本人と判定された発言ブロックのみを抽出する。

出力順は会議録に出現した順とする。

### 9.2 質疑全文

本人発言ブロックを起点に、その次の本人発言が始まるまでの発言群を 1 セットとして抽出する。

例:

1. さんのへ委員
2. 担当部長
3. さんのへ委員
4. 担当部長
5. さんのへ委員

この場合は 1 から 5 までを連続した質疑として出力する。

ただし次の条件で打ち切る。

- 会議進行専用の発言が連続し、質疑が終了したと判断できる場合
- 採決や散会など明確な議事進行に入った場合
- 本人発言と無関係な別議員の発言へ移った場合

### 9.3 本会議の討論

本会議で本人の長い討論が 1 ブロックのみ存在する場合は、
その 1 ブロックをそのまま採用する。


## 10. 整形ルール

### 10.1 タイトル生成

会議録の文書情報から以下形式で生成する。

- `2025年（令和7年）3月18日　厚生委員会`
- `2024年（令和6年）9月26日　第3回定例会（第13号）`

和暦は MVP では次のいずれかにする。

- 実装簡易化のため西暦だけでも可
- 余裕があれば西暦から和暦を自動算出

公開サイトの見た目に寄せるなら、最終的には西暦と和暦の併記が望ましい。

### 10.2 カテゴリ候補生成

会議名からカテゴリ候補を返す。

例:

- `厚生委員会`
- `文教委員会`
- `定例会`
- `予算特別委員会`

### 10.3 段落整形

会議録本文の `<br>` 区切りをもとに段落を維持する。

- 発言者名は先頭へ出す
- 本文中の改行は意味を保つ
- 連続空白や装飾目的の空行は整理する


## 11. 出力仕様

### 11.1 WordPress 用 HTML

最優先の出力形式。

#### 出力例

```html
<p><strong>◯さんのへ委員</strong><br>既にほかの議員から質疑が行われているんですけれども、私もまずは無痛分娩の費用助成について伺います。<br>先日の保健医療局の予算審査でも、無痛分娩費用助成について質疑をさせていただきましたが、本事業は何件分の助成を想定した予算なのか、また、その件数の根拠となる考え方について改めて伺います。</p>

<p><strong>◯瀬川子供・子育て施策推進担当部長</strong><br>令和七年度予算案に、九千五百件の無痛分娩費用助成に係る経費を計上しております。</p>
```

ルール:

- 1 発言ブロックを 1 つの `<p>` にする
- 発言者名は `<strong>` で囲む
- 発言本文は `<br>` で改行を保持する

### 11.2 Markdown

#### 出力例

```md
**◯さんのへ委員**  
既にほかの議員から質疑が行われているんですけれども、私もまずは無痛分娩の費用助成について伺います。  
先日の保健医療局の予算審査でも、無痛分娩費用助成について質疑をさせていただきましたが、本事業は何件分の助成を想定した予算なのか、また、その件数の根拠となる考え方について改めて伺います。
```

### 11.3 プレーンテキスト

#### 出力例

```text
◯さんのへ委員
既にほかの議員から質疑が行われているんですけれども、私もまずは無痛分娩の費用助成について伺います。

```


## 12. エラー表示

ユーザーに返すエラーはわかりやすくする。

- `会議録ページを取得できませんでした`
- `会議録本文ページではない可能性があります`
- `さんのへあや議員の発言が見つかりませんでした`
- `ページ構造が想定と異なり、抽出に失敗しました`


## 13. MVP の実装方針

### 13.1 バックエンド

候補:

- Python
  - `requests`
  - `beautifulsoup4`
  - `lxml`

理由:

- HTML 解析がしやすい
- 文字列整形が簡単
- 今後の自動巡回にも広げやすい

### 13.2 フロントエンド

MVP は以下のどちらかで十分。

- 最短: CLI
- 実用性重視: ローカル Web UI

おすすめはローカル Web UI。
スタッフが使うなら、URL を貼ってコピーボタンを押せる方が明らかに運用しやすい。

最小構成の候補:

- `Flask`
- `FastAPI + Jinja`
- `Streamlit`

最短で形にするなら `Streamlit` が有力。

## 15. MVP 完成条件

以下を満たせば MVP 完成とする。

- 本文ページ URL を入力すると抽出できる
- `質疑全文` と `本人発言のみ` の両方が動く
- 主要な会議種別で動く
  - 定例会
  - 常任委員会
  - 特別委員会
- WordPress 用 HTML をコピーして、そのまま貼り付けられる
- さんのへあや公式サイトの既存記事に近い見た目で出力できる


## 16. 受け入れテスト

最低限、以下のケースで確認する。

1. 本会議の討論記事を 1 件
2. 委員会の質疑記事を 1 件
3. 本人発言が複数回ある会議を 1 件
4. 本人発言がない会議を 1 件

確認ポイント:

- タイトルが正しい
- 本人判定が正しい
- 答弁の紐づけが崩れていない
- 進行発言の除外が過不足ない
- WordPress 貼り付け時に体裁が崩れない


## 17. 次フェーズ

MVP 後に優先度が高い順で追加する。

1. 会議名や日付からの検索
2. 新着会議録の自動検知
3. WordPress 下書き生成
4. 公開済み記事との重複チェック
5. 要約文と SNS 投稿文の生成


## 18. 実装時の判断

実装の初手としては、以下を推奨する。

1. まず `CLI` の抽出ロジックを作る
2. 抽出精度を 3 から 5 本の実データで確認する
3. その後で `Web UI` を載せる

理由:

- 一番難しいのは UI ではなく抽出精度
- URL 入力と HTML 出力のコアが先に固まる
- UI は後から載せ替えやすい
